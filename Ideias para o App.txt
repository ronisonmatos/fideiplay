Boa pergunta de arquitetura. Depende do estágio do projeto:

---
Agora (fase inicial): dados no app está correto

Para o que o FideiPlay é hoje — conteúdo fixo, sem usuários reais, sem backend — manter as perguntas hardcoded no app é a escolha certa. É simples, funciona offline, sem custo.

---
Quando crescer: separar conteúdo do código

As boas práticas dizem que conteúdo que muda com frequência não deve morar no código. Perguntas de quiz são conteúdo, não lógica. Quando isso se torna problema:

- Adicionar 10 perguntas novas exige publicar um novo APK
- Traduzir o app vira um pesadelo
- Não dá para corrigir uma pergunta errada sem atualizar o app

Opção recomendada para vocês: Supabase

É gratuito no tier inicial, tem banco PostgreSQL, API REST automática e autenticação pronta.

tabela: quiz_questions
  id, game_id, question, options (json), correct_index, saint, active

tabela: stop_categories
  id, label, emoji, hints (json)

tabela: bible_verses
  id, words (json), reference, options (json)

O app busca as perguntas na inicialização e cacheia localmente com AsyncStorage. Se não tiver internet, usa o cache. Se não tiver cache, usa um fallback hardcoded mínimo.

---
Caminho incremental (sem reescrever tudo agora)

Fase 1 (hoje)     → dados no app, funciona offline, zero custo
Fase 2 (crescendo) → mover perguntas para JSON em CDN ou Supabase
                     app busca na abertura + cache local
Fase 3 (escala)   → autenticação real, ranking online, perfil salvo

---
Recomendação prática: por ora fica como está. Quando você quiser adicionar um novo pacote de perguntas sem publicar APK, aí vale migrar para Supabase. O código dos jogos não muda quase nada — só troca o array hardcoded por um fetch.

Pronto. O fluxo completo para lançar uma versão agora é:

# 1. merge develop → master (seu fluxo normal)
git checkout master
git merge develop
git push origin master

# 2. criar e subir a tag
git tag v1.1.0
git push origin v1.1.0

Isso dispara o build automaticamente. O artefato gerado vai se chamar santosplay-1.1.0.

Resumo do que cada coisa controla:

┌─────────────────────────┬────────────────┬──────────────────────┐
│                         │ Quem atualiza  │         Como         │
├─────────────────────────┼────────────────┼──────────────────────┤
│ versionName (ex: 1.1.0) │ Você           │ Criando a tag v1.1.0 │
├─────────────────────────┼────────────────┼──────────────────────┤
│ versionCode (ex: 48)    │ EAS automático │ A cada build         │
└─────────────────────────┴────────────────┴──────────────────────┘
